# Babio V2 Roadmap Synthesis

Дата: 2026-07-02

Источник: текущий Babio MVP emulator, консультация Anti, консультация Grok, собственная продуктово-техническая оценка.

## Короткий вердикт

Babio V2 лучше делать не как production-приложение и не как еще один статичный эмулятор. Правильная форма следующего шага: интерактивный PWA-прототип, который:

1. выглядит и ощущается как полноценное приложение для мамы;
2. хранит профиль ребенка, логи и заметки локально;
3. умеет проходить реальный core loop: profile -> log -> ask -> calm step -> notes;
4. сохраняет режим `/studio` и `/record`, чтобы быстро снимать TikTok/Reels/UGC-сценарии;
5. не заходит в медицинскую диагностику, свободный AI-чат, backend, auth и платежи на этой стадии.

## Что сказали Anti и Grok

Anti предложил более строгую архитектурную рамку:

- V2 = Product-Led Interactive Prototype / PWA Sandbox.
- Данные сначала держим в `localStorage`, без backend.
- Нужен Emergency Gateway для опасных запросов.
- Нужен локальный rules engine, а LLM можно подключать позже через защищенный JSON-proxy.
- Не надо делать production app до валидации доверия, сценариев и безопасности.

Grok предложил более продуктовую рамку:

- V2 = enriched product prototype + recording emulator.
- Главная петля: baby profile -> persistent log -> structured ask -> save note.
- Нужно 12-15 scripted moments, чтобы приложение выглядело живым.
- Первый сильный slice: "плохая ночь -> log -> ask -> один спокойный шаг".
- Не добавлять real AI, auth, backend, community, paywall, deep analytics.

Моя оценка: оба вывода совпадают в главном. Делаем интерактивный прототип с настоящими действиями внутри приложения, но с контролируемой логикой и безопасным scripted content.

## Продуктовая позиция V2

Babio не должен становиться "трекером всего" или "медицинским чат-ботом". Его сильная идея:

> У мамы есть тревога и противоречивые советы. Babio берет контекст ребенка и дает один спокойный следующий шаг.

Значит, все функции должны проходить фильтр:

1. Это снижает тревогу прямо сейчас?
2. Это дает один понятный следующий шаг?
3. Это помогает записать контекст для себя или педиатра?
4. Это безопасно и не звучит как диагноз?
5. Это можно красиво снять в 25-35 секунд?

## Что добавить в V2

### 1. Baby Profile

Профиль ребенка: имя, возраст, примерный режим, питание, короткие настройки. Это даст ощущение персонализации и позволит менять copy по возрасту.

Acceptance:

- Home показывает имя и возраст ребенка.
- Flow и Ask используют имя/возраст в тексте.
- Есть demo preset для записи роликов.

### 2. Local Persistent State

`localStorage` для профиля, логов, заметок, последнего Ask и статуса onboarding.

Acceptance:

- Перезагрузка страницы не сбрасывает профиль, логи и заметки.
- Можно очистить demo state из Studio.
- Есть версия схемы данных для будущей миграции.

### 3. Quick Log

Быстрый лог сна, кормления, подгузника, плача/комфорта. Не глубокий baby tracker, а минимум контекста для следующего шага.

Acceptance:

- Запись создается за 1-2 тапа.
- Лог появляется на Home.
- Из записи можно нажать "Спросить про это".

### 4. Structured Ask

Не свободный чат, а управляемый ответ:

- что мы учли;
- что попробовать сейчас;
- что наблюдать;
- что записать;
- когда обратиться к педиатру.

Acceptance:

- Любой ответ имеет одинаковую безопасную структуру.
- Нет диагнозов, обещаний и слов "исправим/вылечим".
- Опасные слова переводят в safety route.

### 5. Notes / Pediatrician Prep

Заметки превращают хаос в короткую сводку для врача или для себя.

Acceptance:

- Ask-result можно сохранить в Notes.
- Notes собирают последние логи и вопросы.
- Можно скопировать summary текстом.

### 6. Scenario Library

Нужно расширить текущие 5 flows до 15-20 безопасных сценариев.

Приоритетные группы:

- сон и ночь;
- кормление;
- плач и успокоение;
- тело/комфорт без диагноза;
- рутина дня;
- подготовка к педиатру;
- эмоциональная поддержка мамы.

### 7. Studio 2.0

Studio должен стать центром сценариев для креативов.

Acceptance:

- Фильтры по моментам: sleep, feeding, comfort, routine, pediatrician.
- Preset state для каждого flow.
- Настройки формата: 9:16, 4:5, 1:1.
- Record mode сохраняет чистую зеленую сцену без подложки.

## Что не добавлять сейчас

1. Свободный AI-чат.
2. Диагностику симптомов.
3. Медикаменты, дозировки, схемы лечения.
4. Backend/auth/accounts.
5. Платежи и подписку.
6. Community/forum.
7. Growth charts и глубокую аналитику.
8. Native iOS/Android.
9. Многоязычность.
10. Cry translator или распознавание плача.

Причина простая: это либо риск, либо отвлечение от главной гипотезы, либо дорого для текущей стадии.

## Стадии реализации

### Stage 0 — Зафиксировать текущий MVP

Цель: не сломать то, что уже работает как эмулятор для записи.

Что сделать:

- оставить текущие `/studio`, `/flow/:flowId`, `/record/:flowId`;
- зафиксировать текущие flows как v1 fixtures;
- расширить content validator под новые unsafe patterns;
- описать flow schema v2.

Готово, когда:

- старые 5 сценариев открываются;
- record mode работает;
- validator проходит.

### Stage 1 — Первый интерактивный core loop

Цель: сделать приложение живым без backend.

Что сделать:

- Baby Profile;
- localStorage store;
- Quick Log;
- Ask result из локальных правил;
- Save to Notes.

Первый slice: "плохая ночь -> log -> ask -> один спокойный шаг".

Готово, когда:

- пользователь вручную проходит путь меньше чем за 90 секунд;
- после reload все данные остаются;
- этот же путь можно открыть в record mode и снять за 25-35 секунд.

### Stage 2 — Scenario Library

Цель: добавить глубину и рекламную вариативность.

Что сделать:

- 12-15 новых scripted scenarios;
- теги сценариев;
- search/filter в Studio;
- presets для записи.

Готово, когда:

- в Studio есть минимум 15 flows;
- каждый flow имеет safe disclaimer;
- каждый flow снимается в 9:16 без ручной настройки;
- unsafe copy validator не находит диагнозов, гарантий и медицинских обещаний.

### Stage 3 — Safety Gateway

Цель: защитить доверие и не давать приложению играть роль врача.

Что сделать:

- список red flags;
- экран Emergency Gateway;
- мягкий warning для неэкстренных, но медицински чувствительных запросов;
- отдельные templates для "позвонить педиатру" и "что записать перед звонком".

Готово, когда:

- "задыхается", "кровь", "судороги", "температура 38 у 2 месяца" не дают обычный совет;
- пользователь видит короткий и ясный safety route;
- record mode может снимать safety сценарий без паники и без юридически опасного текста.

### Stage 4 — Daily Brief + Pediatrician Export

Цель: дать ощущение, что Babio помогает не только в моменте, но и собирает память.

Что сделать:

- morning/day brief из последних логов;
- copy-to-clipboard summary;
- doctor visit prep flow;
- notes grouping by day.

Готово, когда:

- последние 24-48 часов превращаются в короткую сводку;
- summary можно скопировать;
- есть отдельный сценарий "прием через 2 дня".

### Stage 5 — Pilot Readiness

Цель: подготовить не массовый запуск, а закрытую проверку.

Что сделать:

- PWA manifest;
- basic offline shell;
- lightweight analytics stubs;
- feedback prompt после Ask;
- user test script.

Готово, когда:

- приложение ставится как PWA;
- события `profile_completed`, `log_entry_created`, `ask_completed`, `note_saved` логируются;
- можно провести тест на 5-10 мамах без ручного объяснения каждого экрана.

### Stage 6 — AI Hybrid только после проверки

Цель: подключать LLM только если scripted engine уже доказал полезность.

Что сделать позже:

- serverless proxy;
- JSON schema для GuidanceResult;
- local safety filter до запроса в LLM;
- fallback на local rules;
- latency target до 2 секунд.

Это не нужно для первого V2.

## Первый вертикальный срез

Название: "Night Reset: woke again".

Путь:

1. Profile preset: Emma, 9 weeks.
2. Home показывает ночь/усталость и последний сон.
3. Quick Log: "woke at 2:40, awake 45 min".
4. Tap: "Ask about this".
5. Ask открывается с prefilled context.
6. Result:
   - "We used: age + last wake + awake window";
   - "Try now: dim room, calm feed/check, reset timer";
   - "Watch: repeats 3 nights, feeding pattern, wet diapers";
   - "Save for pediatrician";
   - "Call if fever, breathing trouble, unusual lethargy, dehydration signs".
7. Save to Notes.
8. Notes показывает short summary.
9. Record mode проигрывает этот путь автоматически.

Почему этот срез лучший:

- он усиливает уже существующий sleep/woke-up сценарий;
- это самый эмоционально сильный paid social moment;
- он проверяет реальную продуктовую гипотезу, а не только красивый UI;
- он не требует backend и AI;
- он дает основу для всех следующих сценариев.

## Решение по направлению

Рекомендация: начинать Stage 1 с локального deterministic engine. Не подключать AI и backend, пока не появятся 15-20 проверенных сценариев, safety grammar и первые пользовательские реакции.

Технически следующий шаг:

1. Ввести `src/types/babio.ts` для Profile, LogEntry, Note, GuidanceResult, FlowV2.
2. Добавить `src/state/useBabioStore.ts` с localStorage и versioning.
3. Добавить `src/data/guidanceRules.ts` с локальными правилами.
4. Переподключить `/app/home`, `/app/log`, `/app/ask`, `/app/notes` к store.
5. Добавить первый flow V2 `night-reset-woke-again`.
6. Обновить Studio, чтобы показывать V1/V2 flows и presets.
7. Расширить validator.
